# Workspacer v2 — Product & Technical Spec

## 1. What it is

A local-first multi-agent coding cockpit. One Electron app, backed by claudemon (Rust), that gives a single developer command over N parallel Claude Code sessions.

**Core thesis:** the unit of work is not "a terminal" — it's "a decision the human needs to make." Workspacer v2 organizes the UI around an *inbox of decisions*, with sessions as the medium through which work gets done and the terminal as a detail view for the cases where direct intervention is needed.

## 2. What it is not

- Not a Cursor/VS Code replacement. It does not edit code as the primary action; it supervises agents that edit code.
- Not a Conductor clone. It runs on Linux first, is Wayland-native, and keyboard-driven.
- Not a tmux replacement. It complements tmux/Neovim instead of subsuming them.
- Not cloud. State is local; remote access is via Tailscale, not a SaaS backend.
- Not multi-user. One developer, one machine (with optional phone bridge).

## 3. Audience and constraints

- Senior full-stack developer, comfortable in Linux/Wayland/Hyprland, Neovim, tmux.
- Runs ≥3 parallel Claude Code sessions on most working days; peak 8.
- Has built v1 (PTY-attached Electron + Rust claudemon) and knows the substrate works.
- Values keyboard ergonomics, low-latency triage, and not having to context-switch between five tools.

## 4. The three core user flows

### Flow A — Triage (the dominant flow)

1. User glances at Workspacer. Sees inbox at L1 with N items.
2. Scans top 3-5 items by summary line in <5 seconds.
3. Triages: archives 2, snoozes 1, drills into 2.
4. For each drilled item: reads the context paragraph at L2, makes a one-key decision (approve/deny/reply), moves on.
5. Closes Workspacer. Returns when next batch arrives.

**Success criterion:** triage time per item averages <10 seconds at L1, <30 seconds at L2.

### Flow B — Drive (the focused flow)

1. User opens an inbox item that needs more than a quick decision.
2. Drills L2 → L3 to enter the full session view.
3. Reads transcript, takes the wheel via direct PTY input, makes a substantive change.
4. Hands control back to the agent.
5. Exits to L1 when the session returns to a working state.

**Success criterion:** L3 view loads from L2 in <500ms with full PTY scrollback intact.

### Flow C — Reason (the ephemeral-supervisor flow)

1. User has a question that spans sessions ("did billing-refactor decide on the migration approach?") or requires cross-cutting orchestration ("spawn 3 workers to investigate these flaky tests").
2. Opens the Ask panel (right side of UI, or full-screen via shortcut).
3. Types a natural-language query. Workspacer spawns an ephemeral `claude -p` invocation with the claudemon MCP attached.
4. Supervisor responds — possibly with clickable references to inbox items or session diffs.
5. User clicks a reference and is taken to the relevant L2 view.

**Success criterion:** first-token latency <5 seconds, full response with citations <30 seconds for typical queries.

## 5. The three zoom levels

### L1 — Inbox list

The default landing view. Vertical list. One row per item.

**Row layout (left to right):**
- Unread dot (●), read circle (○), or snoozed/resolved glyph (⊙ / ✓)
- Priority color (red / yellow / blue / gray) on left edge
- Worktree name (truncated to 20 chars)
- State icon (⚠ needs decision, ✗ error, ⏳ stuck, 🤖 working, ✓ done)
- Summary line (8-12 words, regenerated by summarizer)
- Age (`2m`, `1h`, `3d`)
- Tail metadata (`[3 files]` or `[+127 -42]`)

**Sections (collapsible):**
- Needs attention (priority ≥ 70)
- Working (priority 10-69)
- Snoozed until *(grouped by time)*
- Resolved today

**Keymap:**
- `j`/`k` — move
- `Enter` — drill to L2
- `o` — jump to L3 (skip L2)
- `e` — archive (mark resolved)
- `s` — snooze (opens menu: 15m / 1h / 4h / tomorrow / on event)
- `r` — quick reply (opens inline input, sends to session)
- `!` — flag (pins to top, pauses agent)
- `Space` — multi-select
- `a` — bulk archive selected
- `/` — search inbox
- `?` — ask supervisor about selected item

**Empty state:** "Inbox zero. 4 sessions working. Last activity 12s ago." plus a today-summary footer (cost, sessions completed, time saved estimate).

### L2 — Item detail

Modal-style overlay. Full-width, ~70% height. Backdrop dims L1.

**Top:** worktree name, age, state, priority color bar.

**Headline:** the summary, large. The thing being decided.

**Context paragraph:** 2-3 sentences. Regenerated by the summarizer when state changes. This is the v2 magic feature.

**Body (three tabs, default = first):**
1. **Decision** — for permission requests: the proposed tool call, its inputs, and why it might be risky/safe. For errors: the error message and the failing command. For "stuck": the loop pattern detected.
2. **Diff** — git diff for the worktree, rendered with syntax highlighting. Files collapsible.
3. **Transcript** — last 30 messages, role-tagged, tool calls collapsed by default.

**Actions footer (one row, always visible):**
- `[y] approve  [n] deny  [r] reply  [s] snooze  [!] flag  [o] open session  [?] ask  [esc] back`

Buttons only appear when applicable. A "working" session has no `y`/`n`; a "done" session has `[m] merge` and `[k] cleanup`.

**Keymap:** all single keys above. Plus `Tab` cycles tabs, `d` jumps to diff tab from anywhere.

### L3 — Session view

Full-screen takeover. The existing Workspacer pane view, largely as-is.

**Layout:**
- Terminal pane (left, ~65% width) — live PTY render, scrollback
- Side panel (right, ~35%) — collapsible. Tabs: diff, transcript, file tree of worktree, info (model, cost, branch)
- Bottom bar — keymap hints, session name, state, cost-so-far, return-to-L2 hint

**Keymap:**
- `Esc` or `q` — back to L2
- `Ctrl-?` — ask supervisor about this session (opens Ask panel docked right)
- `Ctrl-b` then various — tmux-like prefix for pane operations if the user has split into multiple terminals within the session
- Otherwise: PTY captures all input

**Important constraint:** when focus is in the terminal pane, all keys pass through to the PTY. The keymap above only fires when focus is in the side panel or the bottom bar, or via a global Ctrl-modified shortcut. This is the same model Cursor and VS Code use; it works.

## 6. The Ask panel (Flow C surface)

Lives on the right edge, collapsible. Opens on `?` from anywhere, or `Ctrl-/` globally.

**Layout:**
- Input box at bottom
- Conversation history above (scrollable, persists across app launches)
- Each turn shows: user question, supervisor response (markdown-rendered), token/cost footer, tool-call trace (collapsed)

**Behavior:**
- Each question spawns a fresh `claude -p --mcp-config <claudemon-mcp>` invocation.
- Responses stream in.
- Markdown is rendered properly: code blocks, tables, lists.
- When the response references `session_id: feat-auth` or similar IDs, those are auto-linkified — click opens L2 for that session.
- Tool-call trace is collapsed by default; expanding shows what the supervisor queried.

**Preset commands (buttons above input):**
- `Standup` — "summarize what each session did today, what's blocked, what I should look at first"
- `Triage` — "which inbox items are most urgent and why?"
- `Audit` — "any sessions doing risky things (rm -rf, force pushes, etc.)?"
- `Cost` — "what did I spend today, broken down by session?"

Presets are just curated system prompts. Easy to add more in config.

**Per-item asks:** when invoked from L2 with `?`, the ask is pre-loaded with that session's context ID, so the supervisor doesn't have to discover it. The pre-fill text appears in the input as a draft like *"About feat-auth: <your question>"*.

## 7. Snooze model

Critical for keeping the inbox tractable with 5-8 active agents.

**Snooze options:**
- Time-based: 15m, 1h, 4h, tomorrow 9am, custom
- Event-based:
    - "When next event arrives" — useful for "I'll check back when the agent makes progress"
    - "When session completes" — silence chatter until Stop event
    - "When tests pass" — requires hook integration with test runner

**Snoozed items:**
- Hidden from default L1
- Visible in "Snoozed" section (collapsed by default, click to expand)
- Reawaken triggers move them back to "Needs attention" with an unread dot
- Snooze count is shown in L1 footer: "3 snoozed"

**Auto-snooze:** when user takes an action that resolves the immediate need (e.g., approves a permission), the item auto-snoozes "until next event" rather than fully resolving. Resolution happens on session Stop or explicit archive.

## 8. The summarizer

The feature that separates an inbox from a log.

**What it generates:**
- **Summary line** (L1 row): 8-12 words, present tense, action-oriented. "Wants to run prod migration" not "PermissionRequest event received."
- **Context paragraph** (L2): 2-3 sentences explaining what's happening, why it matters, what's at stake. Written for a senior engineer returning from lunch.

**When it regenerates:**
- New `PermissionRequest` event
- `PostToolUseFailure` or `StopFailure`
- Transition into `Stuck` state
- Transition into `Done` state
- User explicitly requests refresh

**How it works:**
- Spawns `claude -p --bare` with last 30 transcript messages + the triggering event
- Uses Haiku (cheaper, fast enough) by default; user can configure to Sonnet
- Caches result on the item; reads from cache on UI render
- Costs roughly $0.001-0.005 per regeneration

**Failure mode:** if the summarizer call fails or times out (>10s), fall back to a deterministic template ("Tool X requested: <tool_name>"). The UI never blocks on summarization.

## 9. Substrate (claudemon, Rust)

The Electron app is a thin client. Everything load-bearing lives in claudemon.

**claudemon responsibilities (v2 additions in bold):**

- PTY management *(exists)*
- Session spawn/lifecycle *(exists)*
- **Hook HTTP listener (`:8765`)** — receives all Claude Code hook events
- **SQLite event store** — items, sessions, events, snoozes, asks, settings
- **Classifier state machine** — derives item state from event stream
- **MCP server** — exposes fleet to ephemeral supervisors
- **Summarizer worker** — spawns `claude -p` for summary/context generation, writes back to DB
- **IPC server for Workspacer** — Unix socket or local HTTP+SSE, subscriptions and commands

**Workspacer responsibilities:**
- All UI rendering
- IPC client to claudemon
- Local UI state (selection, scroll, theme)
- Spawning the Ask `claude -p` and rendering its stream (could move to claudemon, but Electron handling it keeps the renderer in control of presentation)

**Why this split:** if Workspacer crashes, no data is lost. If you swap Electron for Tauri or a TUI later, the substrate doesn't change. Anyone can build another client (CLI, web view, mobile) against the same MCP/IPC.

## 10. Data model (SQLite)

```sql
-- Sessions: one row per worker session, ever
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,               -- claude session uuid
  name TEXT NOT NULL,                -- worktree name, e.g. "feat-auth"
  project TEXT NOT NULL,             -- repo name
  cwd TEXT NOT NULL,
  worktree_path TEXT NOT NULL,
  branch TEXT,
  base_branch TEXT,
  model TEXT,
  state TEXT NOT NULL,               -- working/idle/needs_input/errored/stuck/done/crashed
  pid INTEGER,
  created_at INTEGER NOT NULL,
  last_event_at INTEGER NOT NULL,
  total_cost_usd REAL DEFAULT 0,
  tool_call_count INTEGER DEFAULT 0
);

-- Events: append-only, every hook event + PTY-derived event
CREATE TABLE events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXT NOT NULL,
  timestamp INTEGER NOT NULL,
  event_type TEXT NOT NULL,          -- PreToolUse, PostToolUse, Stop, etc.
  tool_name TEXT,
  payload_json TEXT NOT NULL,        -- raw event JSON
  FOREIGN KEY (session_id) REFERENCES sessions(id)
);
CREATE INDEX events_session_time ON events(session_id, timestamp DESC);

-- Items: the inbox. 1:many with sessions.
CREATE TABLE items (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  state TEXT NOT NULL,               -- unread/read/snoozed/resolved
  priority INTEGER NOT NULL,         -- 0-100
  kind TEXT NOT NULL,                -- needs_input/error/stuck/done/working_milestone
  summary TEXT,                      -- generated by summarizer
  context_paragraph TEXT,            -- generated by summarizer
  next_action TEXT,                  -- "approve" | "review_diff" | "reply" | etc
  triggering_event_id INTEGER,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  resolved_at INTEGER,
  snoozed_until INTEGER,             -- nullable; epoch seconds
  snoozed_on_event TEXT,             -- "next_event" | "session_done" | "tests_pass"
  flagged INTEGER DEFAULT 0,
  FOREIGN KEY (session_id) REFERENCES sessions(id),
  FOREIGN KEY (triggering_event_id) REFERENCES events(id)
);
CREATE INDEX items_state_priority ON items(state, priority DESC, updated_at DESC);

-- Pending decisions: separate so we can track unresolved permission requests cleanly
CREATE TABLE pending_decisions (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  item_id TEXT,
  kind TEXT NOT NULL,                -- permission_request/idle_prompt/confirmation
  tool_name TEXT,
  tool_input_json TEXT,
  prompt_text TEXT,
  created_at INTEGER NOT NULL,
  resolved_at INTEGER,
  resolution TEXT,                   -- allow/deny/ask_user
  FOREIGN KEY (session_id) REFERENCES sessions(id),
  FOREIGN KEY (item_id) REFERENCES items(id)
);

-- Asks: history of supervisor invocations
CREATE TABLE asks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  scope TEXT,                        -- global/project/session/item
  scope_ref TEXT,                    -- corresponding id if scoped
  question TEXT NOT NULL,
  response TEXT,                     -- streamed in
  tool_calls_json TEXT,              -- array of tool call records
  cost_usd REAL,
  duration_ms INTEGER,
  created_at INTEGER NOT NULL,
  completed_at INTEGER
);

-- FTS5 over transcripts for search_transcripts MCP tool
CREATE VIRTUAL TABLE events_fts USING fts5(
  session_id UNINDEXED, timestamp UNINDEXED, content,
  content='', tokenize='porter unicode61'
);
```

**State transitions:** the classifier is a pure function from `(current_session_state, incoming_event) → (new_session_state, [new_or_updated_items])`. Implemented as a Rust match expression. Tested independently with synthetic event streams.

## 11. Classifier rules (canonical)

| Trigger | New session state | Item created/updated | Priority |
|---|---|---|---|
| `SessionStart` | working | — | — |
| `PermissionRequest` | needs_input | new item, kind=needs_input | 95 |
| `Notification(idle_prompt)` | needs_input | new item, kind=needs_input | 90 |
| `Notification(permission_prompt)` | needs_input | new item, kind=needs_input | 90 |
| `PostToolUseFailure` | errored | new item, kind=error | 80 |
| `StopFailure` | errored | new item, kind=error | 85 |
| Same `(tool_name, hash(tool_input))` 5× in 60s | stuck | new item, kind=stuck | 70 |
| No event for 5min while previously working | stuck | new item, kind=stuck | 60 |
| `PreToolUse` cadence healthy | working | update working item if any | 10 |
| `Stop` (no error) | done | new item, kind=done | 40 |
| `SessionEnd` | (terminal) | resolve all open items for session | — |
| Process exit without `Stop` | crashed | new item, kind=error | 75 |

User actions:
- `approve`/`deny` on a needs_input item → write to `pending_decisions`, resolve item, hook returns the decision
- `snooze` → set `snoozed_until` or `snoozed_on_event`, item exits default view
- `flag` → set `flagged=1`, optionally pause agent (send SIGSTOP)
- `archive` → set `state=resolved` regardless of underlying session state

## 12. IPC protocol (claudemon ↔ Workspacer)

Unix socket at `~/.claudemon/sock` (or HTTP+SSE on localhost if you prefer; pick one and stick with it).

**Message format:** newline-delimited JSON, each message has `type`, `id`, `payload`.

**Workspacer → claudemon (requests):**
- `subscribe { topics: ["items", "events", "asks"], filters: {...} }`
- `list_items { include_snoozed, include_resolved }`
- `get_item { id }`
- `get_session { id, transcript_last_n }`
- `item_action { id, action: "approve"|"deny"|"snooze"|"archive"|"flag"|"reply", payload }`
- `ask { question, scope }` → server spawns supervisor, streams tokens back tagged with the ask id
- `attach_pty { session_id }` → server starts streaming PTY bytes for this session
- `detach_pty { session_id }`
- `input_pty { session_id, bytes }`

**claudemon → Workspacer (events, all carry `subscription_id`):**
- `item_changed { item }` — full item state on any update
- `item_created { item }`
- `item_resolved { id }`
- `session_state { id, state, priority }`
- `event_appended { event }` — fine-grained, for "live feed" UI
- `pty_bytes { session_id, bytes }`
- `ask_token { ask_id, token }` and `ask_complete { ask_id, summary }`

**Backpressure:** PTY streams can be paused per subscription. Workspacer pauses streams for sessions not currently visible.

## 13. Hook configuration

Workspacer ships a one-time setup command that writes `~/.claude/settings.json`:

```json
{
  "hooks": {
    "SessionStart":       [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/session_start"}]}],
    "PreToolUse":         [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/pre_tool"}]}],
    "PostToolUse":        [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/post_tool"}]}],
    "PostToolUseFailure": [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/tool_fail"}]}],
    "PermissionRequest":  [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/permission","blocking":true}]}],
    "Notification":       [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/notification"}]}],
    "Stop":               [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/stop"}]}],
    "StopFailure":        [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/stop_fail"}]}],
    "SessionEnd":         [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/session_end"}]}],
    "SubagentStop":       [{"hooks":[{"type":"http","url":"http://localhost:8765/hook/subagent_stop"}]}]
  }
}
```

**Blocking hooks** (PermissionRequest) wait for claudemon's response. Claudemon checks `pending_decisions` table for a resolution, polling with a 30-second timeout. If user has resolved via Workspacer, return the decision; otherwise, return "ask user" and let Claude Code handle interactively.

**All other hooks** return 200 OK immediately, then process async in `tokio::spawn`.

## 14. Performance budgets

- **Hook intake**: <50ms ack from claudemon. Real work async.
- **L1 render**: <100ms from launch with 100 items in DB.
- **L1 to L2 transition**: <150ms.
- **L2 to L3 transition**: <500ms (PTY attach + scrollback load).
- **Ask first token**: <5s (cold `claude -p` start + first MCP call).
- **Summarizer regeneration**: <8s, non-blocking; UI shows previous summary until new one lands.
- **Memory**: claudemon <200MB resident at 8 active sessions. Workspacer (Electron) <500MB.

## 15. Configuration

Single TOML at `~/.config/workspacer/config.toml`. Sample:

```toml
[claudemon]
socket_path = "~/.claudemon/sock"
hook_listen_addr = "127.0.0.1:8765"
db_path = "~/.claudemon/state.db"

[ui]
theme = "dark"
font_family = "JetBrainsMono Nerd Font"
font_size = 13
default_view = "inbox"             # inbox | sessions | ask
inbox_show_resolved_for_hours = 24

[summarizer]
model = "claude-3-5-haiku-latest"
max_regenerations_per_session_per_hour = 20
fallback_template = "[{state}] {tool_name}: {tool_input_summary}"

[supervisor]
model = "claude-sonnet-4"
default_system_prompt = "..."      # path to file
presets_dir = "~/.config/workspacer/presets"

[remote]
ntfy_topic = "https://ntfy.tailnet.example/workspacer"
push_priority_threshold = 70
push_action_button
```
